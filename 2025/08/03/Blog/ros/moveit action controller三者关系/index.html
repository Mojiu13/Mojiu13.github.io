moveit action controller三者关系

一、先给一句“定锚结论”

MoveIt = 任务 / 规划层（Planning & Task Layer）
Action（FollowJointTrajectory）= 执行接口层（Execution API）
Controller（joint_trajectory_controller）= 实时控制层（RT Control）

MoveIt 不直接控制硬件，
它通过 Action 把“执行意图”交给控制器。

二、整体分层结构（从上到下）

三、MoveIt 在“做什么”，它不做什么

✅ MoveIt 负责的事（高层）

  • 运动学（FK / IK）

  • 路径规划（OMPL / STOMP / CHOMP）

  • 约束处理（姿态、碰撞、路径约束）

  • 时间参数化（TOTG / IPTP）

  • 生成 JointTrajectory

👉 它解决的是：
“怎么走这条路”

此外你可以得出一个非常关键的工程结论：

controller 并不是“聪明地忽略点”，
而是 MoveIt 在接口层已经声明：
“这些点只是时间函数的离散参考。”

❌ MoveIt 不负责的事

  • 不 1ms 级循环

  • 不 PID

  • 不 实时性保证

  • 不 直接碰硬件

  • 不 关心电机状态

👉 它不适合实时控制

四、MoveIt 和 Action 的关系

一句话

MoveIt 是 FollowJointTrajectory 的 Action Client

也就是说：

MoveIt 只知道 Action 协议，
不知道 controller 内部怎么实现。

MoveIt 发什么？

  • trajectory_msgs/JointTrajectory

  • time_from_start

  • path_tolerance / goal_tolerance

MoveIt 等什么？

  • Action 状态：

    • ACTIVE

    • SUCCEEDED

    • ABORTED

    • CANCELED

五、Action 和 Controller 的“边界”

这是一个非常重要的理解点：

Action ≠ Controller update()

Action Controller
非实时 实时
状态机 控制环
生命周期长 周期短
语义层 数值层

👉 Action 是“门面”
👉 Controller 是“引擎”

六、Controller 和 ros2_control 的关系

Controller 属于：

ros2_control 的插件体系

  • Controller = 插件

  • ros2_control = 框架

  • Hardware = 插件

它们构成一个 RT 控制岛。

七、MoveIt 和 ros2_control 的关系（关键）

MoveIt 和 ros2_control 不直接耦合

它们通过 Action 协议解耦：

  • 你可以：

    • 换控制器（position / effort）

    • 换硬件

    • 换执行频率

  • MoveIt 不用改

这就是为什么这个架构能“工业级”。

八、用一句“系统级语言”总结（非常重要）

MoveIt 决定“我想去哪、怎么去”；
Controller 决定“我现在这一毫秒怎么动”；
Action 是两者之间的合同。
