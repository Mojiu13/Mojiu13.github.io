详细代码

下面我严格只讲 joint_trajectory_controller::update() 的 B 段，
不回退基础、不碰 A 段、不提前 C / D 段、不讲 MoveIt 内部。

一句话总览（先给结论）
B 段解决的问题只有一个：
👉 “在当前控制周期 update() 被调用的这一刻，我应该执行轨迹上的哪个‘状态’？”

Block 6｜update() 的 B 段在做什么（工程视角）

一、B 段的输入 / 输出（先对齐心智模型）

B 段已知输入

  • time：当前 ROS 时间（rclcpp::Time）

  • period：本次 update 周期时长（Duration）

  • active_trajectory_：当前正在执行的轨迹（已在 A 段确定）

  • state_current_：从硬件读到的当前关节状态（A 段末尾已读）

B 段要算出的核心东西

  • traj_time：当前轨迹执行时间（time_from_start 语义）

  • state_desired_：在这个 traj_time 下，轨迹“期望”的关节状态

👉 B 段的输出不是“下一点”
👉 而是：“当前时间点在轨迹上的理论状态”

二、当前时间是如何计算的？

1️⃣ 核心变量：trajectory 起始时间

  • 不是轨迹消息里的 header.stamp

  • 是 controller 自己认可的“开始执行时间”

2️⃣ 当前轨迹执行时间（关键公式）

也就是：

traj_time ≡ time_from_start

这一步非常重要：

名字 含义
time 当前 update 调用的真实时间
trajectory_start_time_ 轨迹开始执行的时间
traj_time 轨迹内部坐标系的时间

👉 MoveIt / Action 里所有的 time_from_start，最终都要映射到这个 traj_time

3️⃣ 为什么不用“第几个点”？

因为 controller 是 时间驱动（time-driven），不是 索引驱动（index-driven）

如果你用点索引：

  • update 频率变化 → 轨迹执行快慢不一致

  • 硬件 jitter → 执行抖动

  • 插值无法统一

时间是唯一稳定的自变量

  • controller 可能在 200Hz update

  • 但轨迹点只有 50Hz

👉 插值是 controller 的职责，不是 planner 的

五、为什么 controller 不“严格逐点执行”？

这是你这个 Block 最关键的问题之一。

❌ 错误直觉

“轨迹不是一系列点吗？那就一个点一个点执行不就好了？”

✅ 工程现实

1️⃣ 控制周期 ≠ 轨迹点周期

  • update()：100–1000Hz

  • trajectory points：10–100Hz

不可能一一对应。

2️⃣ 实时控制不能阻塞

如果你“等到点 i 执行完再去 i+1”：

  • 硬件 jitter

  • RT 线程阻塞

  • 控制不稳定

时间驱动是唯一可实时化的方式

3️⃣ 插值才是“真实轨迹”

轨迹点只是约束点（waypoints）
真实执行的是：

连续函数 q(t)，而不是离散点 q[i]

六、B 段在整个 update() 中的地位

你现在可以把 update() 分成这样理解：

👉 B 段是唯一“纯数学 + 时间”的一段
👉 它不关心：

  • goal

  • action

  • MoveIt

  • 硬件接口

七、给你一个“工程级一句话总结”

joint_trajectory_controller 的 B 段不是在“执行点”，
而是在“评估一个时间函数在当前时刻的值”。
