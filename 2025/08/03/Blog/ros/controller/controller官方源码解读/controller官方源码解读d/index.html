Block 8｜update() 的 D 段（Command Interface 写入）

核心一句话目标
controller 已经“算清楚要什么”，D 段只做一件事：
把期望状态，安全、确定、实时地写进 command interface

0️⃣ D 段在整个 update() 中的位置认知（定位，不复述）

  • 输入：

    • 已完成采样的 desired state（position / velocity / effort）

    • controller 当前的 mode / state

  • 输出：

    • command_interfaces_ → hardware

  • D 段不做：

    • 不判断容差

    • 不改轨迹

    • 不关心硬件细节

👉 它是“最后一道闸门”

1️⃣ command interface 的工程模型（controller 视角）

1.1 command interface 是什么（抽象，不是驱动）

  • command_interfaces_ 的真实语义

    • 不是“电机”

    • 不是“总线”

    • 是 对 hardware 的“写权限句柄”

  • controller 只看到：

    • joint 名

    • interface 类型（position / velocity / effort）

    • 一个 set_value(double)

👉 这是 ros2_control 的最小实时契约

1.2 为什么是 vector + handle，而不是 map / class / 智能对象

  • 顺序固定

  • 无动态分配

  • 无查找

  • 可预测 cache 行为

👉 D 段是“实时路径”，一切为了确定性

2️⃣ position / velocity / effort：不是“功能选择”，而是“责任边界选择”

2.1 position command（最常见）

  • controller 责任：

    • 给硬件一个绝对目标

  • hardware 责任：

    • 内部 PID / 插值 / 力控

工程含义

controller = 轨迹时序
hardware = 控制律

如果 controller 试图“更聪明”
❌ 自己算 PID →

  • 双重控制环

  • 实时抖动

  • 安全不可控

2.2 velocity command（中间态）

  • controller 给的是：

    • “此刻你应该动多快”

  • 对实时要求更高：

    • 速度误差 = 立刻发散

工程风险

  • 采样抖动直接反映到关节

  • 上层 jitter → 硬件 jitter

👉 所以 velocity 模式 极少用于 MoveIt 执行

2.3 effort command（最危险）

  • controller 在下达：

    • 力 / 力矩

  • 意味着：

    • controller = 控制律的一部分

为什么 joint_trajectory_controller 默认不推荐？

  • effort 一旦算错：

    • 没有硬件兜底

    • 直接损坏

👉 这是“研究模式接口”，不是工程默认

3️⃣ D 段的真实代码逻辑分层（你接下来会看到的）

3.1 joint → interface → command 的映射

  • command_handles_

  • command_interfaces_

  • index 一一对应

关键设计点

  • 不允许运行期重排

  • 不允许“这个 joint 没有 interface”

3.2 “只写，不判断”的哲学

  • D 段只做：

  • 不做：

    • clamp

    • filter

    • smooth

    • safety check

为什么？

👉 因为这些已经在 B / C 段完成
👉 D 段是“执行，不思考”

4️⃣ 为什么 controller 不关心 hardware 怎么执行

4.1 这是 ros2_control 的系统边界

  • controller：

    • 实时

    • 确定

    • 无副作用

  • hardware：

    • 设备相关

    • 不可预测

    • 允许复杂

边界一旦打破会发生什么？

❌ controller 感知硬件状态
→ update() 阻塞
→ 实时性破坏
→ 全系统抖动

4.2 command interface 是“单向流”

  • state → controller

  • controller → command

  • 永不反向查询

这是实时系统的铁律。

5️⃣ realtime 约束下，D 段的设计红线（工程警戒线）

5.1 D 段绝对禁止的行为

  • ❌ new / delete

  • ❌ std::vector resize

  • ❌ map / unordered_map

  • ❌ string

  • ❌ 日志

  • ❌ 条件分支爆炸

5.2 为什么“不更聪明”反而更安全

你可以把 D 段理解成：

一个“写寄存器循环”

  • 聪明 = 不确定

  • 简单 = 可验证

  • 可预测 = 安全

6️⃣ Block 8 学完后的“能力产出”

**joint_trajectory_controller 的 D 段不是“控制”，
而是“把已经决定好的控制意图，原封不动地交出去”。

最终一句话总结

joint_trajectory_controller 的 D 段不是控制器，
而是一个实时安全的命令交付器；
command interface 之所以极度简陋，
是为了确保 controller 与 hardware 的边界永远清晰、可验证、不可滥用。
