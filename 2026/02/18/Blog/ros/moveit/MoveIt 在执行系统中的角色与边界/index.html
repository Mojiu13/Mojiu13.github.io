Block M1｜MoveIt 在执行系统中的角色与边界（学习大纲）

视角前提：

  • controller = 实时、确定性、状态机 + 容差 + 写接口

  • MoveIt = 非实时、规划态、一次性决策

一、执行系统的三层切分（从 controller 反推）

目标：先建立一个“工程分层地图”，后面所有结论都能落在这张图上

  • 执行系统的三层结构：

    1. Planning 层（MoveIt）

    2. Execution 层（controller）

    3. Physical 层（hardware / drivers）

  • 每一层的时间尺度

  • 每一层对“失败 / 抖动 / 延迟”的容忍度

  • 为什么 controller 的 update() 天然不可能属于 MoveIt

✔ 对应 controller：
update() 的存在本身，就是对“上层非实时”的工程回应

层级 时间观 是否允许回退 是否允许阻塞
MoveIt（Planning） 离散 / 非实时 ✅ 可以 ✅ 可以
Controller（Execution） 连续 / 实时 ❌ 绝对不行 ❌ 绝对不行
Hardware 物理时间 N/A ❌（否则失控）

二、Planning vs Execution 的根本分工（不是功能分工）

目标：解释清楚「为什么 MoveIt 只能 生成轨迹」

  • Planning 层真正解决的是什么问题？

    • 离散决策 vs 连续控制

    • 全局最优 vs 局部稳定

  • Execution 层真正解决的是什么问题？

    • 时间推进

    • 误差积累

    • 异常不可避免

✔ 对应 controller：

  • A 段：状态机不是规划

  • B 段：时间是执行层的唯一主权

  • C 段：容差 = 对现实世界不完美的承认

三、MoveIt 为什么必须对实时执行“无感”

目标：不是“MoveIt 不管”，而是“不能管”

  • MoveIt 若感知实时执行，会违反哪些工程假设？

  • 为什么 MoveIt 不应知道：

    • 当前关节是否抖了一下

    • 是否有单 joint 轻微越界

  • MoveIt 若试图介入 update 级别决策，会发生什么

✔ 对应 controller：

  • sample() 的存在，本身否定了 MoveIt 的实时介入可能

  • monotonic time search 的设计前提

四、哪些问题必须在 MoveIt 解决

目标：明确 MoveIt 的“不可替代职责”

  • 一次性、全局性、不可回滚的问题

  • 对执行层来说太晚的问题

  • 典型例子（只点名，不展开）：

    • 目标可达性

    • 轨迹结构合法性

    • 执行前的约束一致性

✔ 对应 controller：
controller 假设 trajectory 本身是“物理上可执行的”

五、哪些问题必须留给 controller

目标：解释为什么这些问题 MoveIt 碰都不该碰

  • 连续时间中的误差问题

  • 硬件延迟与采样抖动

  • 执行中状态偏移

  • “已经在执行了”的一切情况

✔ 对应 controller：

  • tolerance violated while moving

  • goal time tolerance

  • hold / abort 的触发条件

六、职责混淆会导致的工程灾难（反例推导）

目标：用“如果混在一起会怎样”来反证边界合理性

  • 若 MoveIt 参与实时决策：

    • 执行震荡

    • 上下层互相抢主权

  • 若 controller 试图做规划决策：

    • 状态爆炸

    • 不可验证安全性

  • 最终结果：

    • 系统既不确定、也不安全、也不可维护

✔ 对应 controller：
update() 必须是闭环且自洽的，否则整个执行层崩塌

七、Block M1 的“核心结论

这一段在你学完后我们会一起精炼成一句话

  • MoveIt 只存在于执行开始之前

  • controller 是执行阶段的唯一主权者

  • Planning 与 Execution 的分界线是“是否允许回头”

  • trajectory 是两层之间唯一合法的契约
