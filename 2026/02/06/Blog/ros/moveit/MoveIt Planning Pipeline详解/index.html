Block M2｜MoveIt Planning Pipeline（执行系统接口视角）

主题主线：为什么 MoveIt 只生成 trajectory，而不是直接下发 position

视角前提：
✅ 你已经完全理解 controller 的时间推进 + 采样模型
✅ 你已经明确 controller / hardware 的真实边界
❌ 本 Block 不讨论任何算法细节，只讨论“接口为什么这么设计”

M2-0｜重新画一条“执行导向”的系统边界图（对齐 Block M1）

目标：把 Planning Pipeline 放回你已经理解的执行系统里

  • MoveIt 在整个系统中的“非实时定位”

  • Planning Pipeline 位于：

    • Action 之前

    • Controller 之外

    • Hardware 之上

  • 从 controller 视角反推：

    • controller 真正“需要”的是什么类型的数据？

🔑 本节会把 Planning Pipeline 的输入 / 输出 和
你已经精读的 update() 中 轨迹假设 对齐

M2-1｜Planning Pipeline 的工程定位：它解决的不是“控制”，而是“执行前的确定性”

核心问题：

如果 controller 已经能稳定执行 trajectory，
那 Planning Pipeline 还“必须存在”的工程原因是什么？

本节将拆解：

  • Planning Pipeline 在系统中承担的唯一责任

  • 它解决的是哪一类 controller 不能、也不应该解决的问题

  • 为什么这些问题必须在“时间轴生成之前”完成

对齐点：
controller 的 sample(t) 为什么假设 trajectory 已经是“干净的”

M2-2｜为什么 Planning Pipeline 的最终产物是 RobotTrajectory / JointTrajectory

这是整个 Block 的中心

我们会从 controller 的行为反向推导：

  • controller 的核心假设：

    • trajectory 是 时间参数化的

    • trajectory 是 离散点 + 插值可定义的

  • JointTrajectory 在接口层面提供了什么？

    • 时间单调性

    • 全关节同步快照

    • 插值语义的“合法输入空间”

你会看到：
JointTrajectory 不是“规划结果”，而是“执行接口契约”

M2-3｜为什么 Planning Pipeline 不直接输出 position setpoint

这一节会非常“工程”，不讲任何算法。

我们会逐条否定这种设计：

  • 如果 Planning Pipeline 直接输出 position：

    • controller 的 update() 会被迫承担什么？

    • 时间推进模型会在哪里崩塌？

  • position setpoint 与 controller 的：

    • trajectory_start_time_

    • time_from_start

    • monotonic sampling
      的结构性冲突

结论会非常明确：
直接 position = 把非实时、不确定性塞进实时线程

M2-4｜Planning Pipeline 与 controller 时间采样模型的“天然对齐”

这是你会觉得**“啊，原来如此”**的一节。

核心对齐点：

  • Planning Pipeline 输出：

    • 一条 完整时间轴

  • controller 只做一件事：

    • 在时间轴上 前向采样

我们会逐项对齐：

  • time_from_start ↔ sample_time

  • trajectory segment ↔ monotonic iterator

  • 插值方法 ↔ controller 的职责边界

你会看到：
MoveIt 在接口层已经为 controller 的 update() 铺好路

M2-5｜一句话工程结论（为进入 Block M3 做准备）

本节只做一件事：

  • 用 3～5 条工程级断言总结：

    • 为什么 Planning Pipeline 必须止步于 trajectory

    • 为什么这是一个“最小责任、最大稳定性”的接口设计

这会直接引出后续可学内容，例如：

  • trajectory 时间参数化插件

  • execution pipeline / trajectory adapter

  • multi-controller / multi-mode 执行扩展

Block M2 学习完成判定标准（给你一个自检锚点）

当你学完 Block M2，你应该可以不看代码回答：

  • 为什么 controller 不关心规划算法

  • 为什么 Planning Pipeline 不关心实时性

  • 为什么 JointTrajectory 是两者之间唯一合理的契约

  • 如果让你设计一个新的执行系统，你是否会做同样的选择？为什么？

Block M2｜Planning Pipeline 的接口设计意义

（执行系统接口视角 · 非算法视角）

一、问题定义

在已经完全理解：

  • joint_trajectory_controller::update()

  • Trajectory::sample()

  • 时间推进模型

  • 副作用隔离设计

之后，需要回答一个工程问题：

MoveIt Planning Pipeline 为什么止步于 trajectory？
为什么不直接输出 position setpoint？

二、系统分层（执行导向）

从 controller 视角反推系统结构：

关键边界：

  • Planning Pipeline 不在实时线程

  • Controller 是唯一写 command interface 的模块

  • Trajectory 是非实时 → 实时之间的“接口对象”

三、controller 的设计公理

controller 的核心不是“控制”，而是：

时间驱动的轨迹播放器

其行为模型：

  1. 固定周期 update()

  2. 计算当前时间：

  3. 调用：

  4. 写 command interface

关键假设：

  • 时间单调前进

  • trajectory 先验存在

  • trajectory 不在执行过程中被改写

  • sample 是纯函数

  • 副作用只发生在最后一步

四、Planning Pipeline 的工程职责

Planning Pipeline 的存在不是为了“算路径”，
而是为了解决 controller 永远不该解决的问题：

1️⃣ 全局一致性（一次性确定）

必须在进入实时系统前确定：

  • 起点与终点

  • 连续性

  • 时间轴完整性

  • 单调性

controller 不具备全局视角，只负责“当前时刻”。

2️⃣ 可失败性隔离

Planning Pipeline 可以：

  • 失败

  • 重试

  • 回滚

  • 多次尝试

controller 不能：

  • 不能阻塞

  • 不能“再算一次”

  • 不能在 update() 中失败重来

因此：

所有可失败计算必须在非实时层完成。

3️⃣ 时间结构生成

controller 的运行模型要求：

trajectory 必须是一个时间函数

形式化表达：

这意味着 trajectory 必须具备：

  • time_from_start

  • 单调时间序列

  • 插值语义

Planning Pipeline 的输出正是：

  • RobotTrajectory

  • JointTrajectory

本质：离散时间节点 + 插值定义

4️⃣ 副作用隔离（核心工程原则）

在 ros2_control 中：

写 command interface 是唯一副作用。

因此：

  • 副作用必须集中在 controller

  • 非实时模块不得间接支配实时输出

如果 Planning Pipeline 输出 position setpoint：

  • 实时线程必须承担时间推进

  • 必须处理跳变

  • 必须做平滑

  • 必须做决策

这违反：

实时线程只做“确定性、可界定”的工作

五、为什么不能输出 position setpoint

从 controller 时间模型直接否定：

❌ 1. position 无时间身份

controller 需要：

position 只提供：

无法回答：

  • 这是哪个时间点的状态？

❌ 2. 破坏单调采样假设

controller 使用：

position 可能：

  • 跳变

  • 回退

  • 被临时修正

破坏前向采样模型。

❌ 3. 把时间推进责任塞入实时线程

如果没有 trajectory：

controller 必须：

  • 决定当前目标

  • 处理突变

  • 计算过渡

  • 管理时间连续性

这等价于：

把 Planning 的责任强行放进 controller。

❌ 4. 副作用来源失控

原设计：

错误设计：

pipeline 间接影响实时输出，却不承担实时责任。

这是严重的接口耦合问题。

六、Trajectory 为什么是完美接口对象

trajectory 同时满足：

  • 无副作用

  • 可整体校验

  • 可整体替换

  • 可缓存

  • 可复制

  • 可时间采样

  • 可插值

  • 可在非实时生成

它是：

非实时与实时之间的“安全抽象层”

七、Planning Pipeline 与 controller 的天然对齐

逐项对齐：

controller 需求 trajectory 提供
时间锚点可平移 time_from_start
单调前向采样 单调时间序列
segment 插值 waypoint + 速度语义
无副作用输入 纯数据对象
可整体替换 整条 trajectory

这不是巧合，而是接口级匹配。

八、五条工程断言（最终总结）

① Planning Pipeline 是“时间结构生成模块”，不是控制模块。

② trajectory 是执行意图的时间函数表示。

③ controller 是 trajectory 的播放器，而不是目标决策器。

④ position setpoint 在时间采样模型下是未定义行为。

⑤ 副作用必须锁在 controller，因此 Pipeline 必须止步于 trajectory。

九、一句话终极总结（可面试用）

MoveIt Planning Pipeline 的存在意义，不是为了“给控制器目标位置”，
而是为了在非实时层一次性生成一个完整、单调、可插值的时间函数，
使 controller 只需按时间采样并写入 command interface。

trajectory 是实时系统中唯一合理的接口抽象。

Block M2 状态

✔ 范围内问题已全部闭环
✔ 与 controller update()/sample 行为完全对齐
✔ 未展开算法、IK、PlanningScene、控制内部实现

Block M2 完成。
