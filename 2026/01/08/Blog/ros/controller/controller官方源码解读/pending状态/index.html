pending状态

Mojiu Lv3
  2026-01-08 22:17:58 2026-01-08 22:17:58 Created   2026-01-08 21:53:51 2026-01-08 21:53:51 Updated  

下面我给你一个**“一次性定型”的总结**：
pending 是什么、为什么存在、整个过程中每个关键变量在什么时候变、变成什么。
你可以把这份当成 joint_trajectory_controller 的“pending 状态白皮书”。

一句话总览（先记住）

pending 是 RT/Non-RT 之间的“过渡态协议”：
新 goal 已经到达，但 RT 还没在安全点正式接管之前，
系统必须进入一个“禁止换轨、禁止混用状态”的中间窗口。

一、涉及的核心变量（角色先定清）

变量 所在层 含义
rt_has_pending_goal_ RT-safe 标志 是否有新 goal 等待接管
rt_active_goal_ RealtimeBuffer 当前 RT 认可的 active goal 句柄
active_goal update 局部快照 本次 update 看到的 active goal（可为空）
new_trajectory_msg_ RealtimeBuffer Non-RT 写入的新轨迹消息
current_trajectory_ RT 对象 当前可采样执行的轨迹
state_current_ RT 快照 当前 update 的真实状态（t=0）

二、完整 pending 流程（时间线版）

下面是一次典型的“旧 goal → 新 goal”切换全过程。

阶段 0｜稳定执行旧 goal（正常态）

状态：

  • rt_has_pending_goal_ = false

  • rt_active_goal_ = 旧 goal（非空）

  • current_trajectory_ = 旧轨迹

语义：

RT 正常采样、执行旧轨迹。

阶段 1｜Non-RT 收到新 Action goal（pending 产生）

Non-RT（Action 回调）做的事：

  • 接收到新 goal

  • 不直接改 RT 状态

  • 写入：

    • rt_has_pending_goal_ = true

    • new_trajectory_msg_ = 新轨迹

此刻状态：

  • rt_has_pending_goal_ = true

  • rt_active_goal_ 仍然是旧 goal

语义：

新任务来了，但 RT 还没点头。

阶段 2｜RT 进入“切换窗口”（危险态）

在某次 update() 中，旧 goal 结束 / 被 reset，但新 goal 还没激活。

RT 中发生：

  • rt_active_goal_.reset()
    👉 active_goal == nullptr

  • rt_has_pending_goal_ == true

此刻关键组合：

🚨 这是 pending 的核心时刻

系统明确声明：

“新 goal 已到，但当前没有任何 active goal。”

代码中的防线：

👉 禁止换轨
👉 禁止让新轨迹提前生效

阶段 3｜RT 在安全点接管新 goal（pending → active）

在随后的某次 update 的安全位置：

RT 做的事：

  • rt_active_goal_ = pending_goal

  • rt_has_pending_goal_ = false

此刻状态恢复稳定：

  • active_goal != nullptr

  • 没有 pending

语义：

RT 正式接受新任务。

  • current_trajectory_ 切换为新轨迹

  • 后续 update 正常采样执行

三、为什么必须把 active_goal 设为空？

这是 pending 设计的灵魂点。

active_goal = nullptr 不是“没 goal”，
而是一个“同步屏障信号”：
告诉 RT——现在不是稳定态，任何轨迹切换都必须暂停。

如果不这样做，会发生：

  • 新轨迹已切

  • 旧 goal / 新 goal 语义混用

  • 成功 / 失败判断错乱
    👉 竞态条件

四、用一句“工程师级因果链”总结

五、pending 的本质（最重要一句）

pending 不是状态机里的一个“状态”，
而是 RT/Non-RT 之间的“同步协议”。

它存在的唯一目的：

防止 goal 与 trajectory 在时间上错位，从而消除竞态。

六、你现在已经“完全理解”的标志

如果我现在问你：

为什么 (rt_has_pending_goal_ && !active_goal) 时必须禁止换轨？

你能自然回答：

因为这是 goal 切换的中间窗口，
active 被刻意清空作为同步屏障，
此时换轨会导致 goal 与 trajectory 不一致，属于竞态。

如果你能这么说——
pending 这块你已经 100% 吃透了。
