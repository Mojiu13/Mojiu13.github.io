Block M3b｜MoveIt 侧 trajectory 构造机制（学习大纲）

主线问题只有一个：
controller 里 sample(traj_time) 用的那条轨迹，MoveIt 是怎么一步步“喂”出来的？

一、MoveIt 中「trajectory」的三层结构（非常关键）

目的：先统一“名词 ↔ 实体 ↔ 数据结构”的映射

1️⃣ 抽象层：RobotTrajectory

  • 在 MoveIt 内部的统一轨迹容器

  • 同时承载：

    • 关节轨迹（Joint）

    • 笛卡尔轨迹（可选）

  • 此时还不是 ROS 消息

RobotTrajectory 作为中间层，让 MoveIt 可以：

  • 规划阶段只产出 path（无时间）

  • 后处理阶段再做 时间参数化（产生相对时间轴）

  • 最终导出 JointTrajectory 时，time_from_start 表示：

    • 相对于轨迹起点的相对时间

    • 而不是某个外部 wall clock 的绝对时间

👉 这就完美适配 controller B 段的模型：
controller 用自己的 trajectory_start_time 把相对时间轴“挂到”当前 now 上，然后在任意时刻采样。
👉 对应 controller：

controller 永远看不到 RobotTrajectory，只看到它“被导出”的结果

  • Qs：为什么 time_from_start 一定要在“RobotTrajectory 已经完整存在之后”才生成，而不能边规划边生成
  • 如果规划器直接输出 JointTrajectoryPoint[]，会立刻产生三个问题： planner 必须决定每个点的时间 planner 必须假设最大速度 / 加速度 planner 必须知道 controller 的执行语义 👉 全部都是 planner 不该知道的东西
  • AS：如果规划器直接生成 JointTrajectory，它就必须绑定执行侧的时间模型、动力学约束和控制语义，这是一个架构级职责越界。

2️⃣ 规划结果层：trajectory_msgs/JointTrajectory

  • RobotTrajectory → 导出成 ROS 标准消息

  • 包含：

    • joint_names

    • points[]（positions / velocities / accelerations / time_from_start）

👉 对应 controller：

controller 的 active_trajectory_ 就是从这里来的

导出时，MoveIt 实际做了哪三件关键事情

3️⃣ 执行层：FollowJointTrajectory Action

  • JointTrajectory 被包装进 Action goal

  • 发送给 joint_trajectory_controller

👉 对应 controller：

A 段接收 → B 段 sample → C/D 段校验与输出

二、RobotTrajectory 是如何“被生成”的（不进插件细节）

目标：只看结果如何被写入容器，不看“怎么算出来的”

4️⃣ 规划阶段：谁在往 RobotTrajectory 里写点？

  • OMPL / CHOMP 等 planner 只负责产出“路径”

  • 路径是：

    • 离散的关节状态序列

    • 没有时间概念

👉 关键结论（先记住）：

规划阶段 ≠ 时间阶段

三、时间是“后加的”，不是规划出来的（核心）

这是整个 Block M3b 的核心思想

5️⃣ 时间参数化（Time Parameterization）

  • 在 MoveIt 中是一个独立步骤

  • 输入：

    • RobotTrajectory（只有路径）

  • 输出：

    • 给每个 waypoint 分配：

      • velocity

      • acceleration

      • time_from_start

👉 重要：

time_from_start 在 MoveIt 内部完成，而不是 controller

6️⃣ time_from_start 的工程含义（为 controller 服务）

  • 是一个严格单调递增的时间轴

  • 描述的是：

    • “从轨迹起点开始已经过了多久”

👉 对应 controller B 段：

四、RobotTrajectory → JointTrajectory 的“导出瞬间”

这是 MoveIt 与 ros2_control 的唯一真实接口

7️⃣ 导出规则（非常重要）

  • RobotTrajectory 内部数据被“拍平”为：

    • JointTrajectoryPoint[i].time_from_start

  • MoveIt 不关心 controller update 频率

  • 只保证：

    • 时间轴连续

    • 点序有序

👉 对应 controller：

controller 不逐点执行，而是“时间函数求值”

五、为什么它们“刚好适配” controller 的采样模型

用你 Block 6 的理解来反证

8️⃣ MoveIt 假设了什么？

  • controller 是：

    • 周期性 update

    • 基于时间采样

  • controller 会：

    • 在任意 traj_time 下插值

9️⃣ controller 又假设了什么？

  • trajectory 是：

    • 时间参数化完成的

    • 不要求逐点对齐 update tick

👉 两边在“时间连续性”上达成隐式契约

六、工程级闭环总结（你学完应该能一句话说清）

我们最后要收敛到一句话：

MoveIt 负责生成“带时间轴的期望运动函数”，
controller 只是在实时循环里对这个函数做采样。

✔️ 学习节奏建议

  • 每一节都 反向指 controller 的 B 段

  • 不引入新系统，只验证你已经啃过的模型

核心问题讨论

1️⃣ RobotTrajectory 为什么存在

一句话定论：

RobotTrajectory 的存在，是为了把“路径问题”和“时间 / 执行问题”彻底解耦。

工程含义：

  • 规划器（OMPL / CHOMP）
    👉 只负责：从 A 到 B 有没有一条连续可行的路径

  • controller
    👉 只负责：在实时循环中，某个时间点该执行什么状态

如果没有 RobotTrajectory：

  • planner 就必须：

    • 决定时间

    • 决定速度 / 加速度

    • 理解 controller 的执行模型

👉 这是架构级职责越界

所以：

RobotTrajectory 是 MoveIt 内部的“中间态轨迹容器”，
用来承载“还没绑定时间语义的运动结果”。

2️⃣ time_from_start 是谁、在什么时候、因为什么加的

一句话定论：

time_from_start 是 MoveIt 在“时间参数化阶段”加的，
不是规划器加的，也不是 controller 加的。

✔️ 谁加的

  • MoveIt 的 时间参数化模块

  • 输入：RobotTrajectory（只有路径）

  • 输出：RobotTrajectory（路径 + 时间）

✔️ 什么时候加

  • 规划完成之后

  • 轨迹即将进入执行之前

✔️ 为什么加

为了满足 controller 的核心假设：

也就是说：

time_from_start 的唯一目的，是把轨迹变成“可按时间采样的函数”。

3️⃣ 为什么 controller 不逐点执行

一句话定论：

因为 JointTrajectoryPoint 在语义上不是“指令列表”，
而是“连续时间轨迹的离散采样点”。

工程事实：

  • MoveIt 在导出 JointTrajectory 时已经声明：

    • 时间轴是连续的

    • 点之间允许插值

  • controller 的职责是：

    • 在任意 traj_time 下求期望状态

所以：

controller 既没有义务，也没有能力
“严格在某个 update tick 执行某一个点”。

4️⃣ 为什么 controller 的 B 段 设计得“理直气壮”

你在 Block 6 精读过的这段代码：

一句话定论：

因为 MoveIt 已经保证：
controller 接收到的是一条“时间参数化完成的轨迹函数”。

所以 B 段可以理直气壮地假设：

  • 时间是连续的

  • update 频率可以抖动

  • 任意时间点都可以 sample

这不是 controller “偷懒”，而是：

MoveIt 与 ros2_control 之间的隐式工程契约。

🧩 四点合在一起的“终极一句话”

MoveIt 的职责是生成“带时间轴的期望运动函数”，
controller 的职责是在实时循环中，对这个函数做时间采样并执行。

MoveIt → ros2_control 时间模型（工程结论）

1. MoveIt 输出的本质

MoveIt 输出的不是“逐点执行指令”，
而是一个 带时间轴的期望运动函数 f(t)。

2. RobotTrajectory 的角色

RobotTrajectory 是 MoveIt 内部的中间轨迹容器：

  • 承载规划得到的路径
  • 在执行前独立进行时间参数化
  • 防止规划器耦合 controller 的时间/执行语义

3. time_from_start 的来源

time_from_start：

  • 由 MoveIt 在时间参数化阶段生成
  • 不是 planner 的结果
  • 也不是 controller 决定的
    其目的只有一个：
    → 让轨迹可被按时间采样

4. controller 的执行模型

controller 执行逻辑：
traj_time = now - trajectory_start_time
state = sample(traj_time)

因此：

  • controller 执行的是时间函数
  • JointTrajectoryPoint 是离散采样点，不是指令列表
