学习大纲

主题：joint_trajectory_controller 中 trajectory sampling 的工程必然性

核心问题只有一个：
为什么 sample / interpolate 必须这样写，否则 realtime 执行会出什么事故？

0️⃣ 先对齐一个“工程视角前提”（非常重要）

在进入任何代码前，你需要带着这三个前提去读：

  • 控制器不是规划器
    它只做一件事：

    在 当前控制周期 给出 唯一可执行的期望状态

  • trajectory sampling 是 realtime-safe 的“最后一道解释层”
    上游任何东西都不可信：

    • 时间可能抖

    • 轨迹可能不连续

    • 执行周期可能跳变

  • sampling 的职责不是“算得准”，而是“不炸”
    精度 < 安全 < 可预测性

1️⃣ 轨迹时间定位模型（Time → Trajectory）

要读的逻辑块

  • traj_time = current_time - trajectory_start_time

  • 为什么 不用 ROS time 直接对比 time_from_start

你需要搞清的问题

  • controller 内部认的“时间基准”是什么？

  • trajectory_start_time 为什么必须被 latch？

如果不这样写的工程风险

  • 控制周期抖动 → 采样点前后跳

  • ROS time 回拨 / pause → traj_time 倒退

  • 同一轨迹在不同周期被“重新解释”

2️⃣ 越界时间的硬处理策略（traj_time < 0 / > end）

要读的逻辑块

  • if (traj_time < 0)

  • if (traj_time > trajectory_duration)

你需要搞清的问题

  • 为什么 负时间不报错，而是 clamp？

  • 为什么 超出 end 不是插值，而是 final point

如果不这样写的工程风险

  • 负时间 → 插值分母为负 → 速度方向反转

  • end 后继续插值 → 末端抖动 / 越位

  • 控制器在“成功点”附近永不收敛

3️⃣ 区间查找：为什么不是每次都从头找

要读的逻辑块

  • sample_index_ / cached index

  • index forward-only 推进

  • while (time_from_start < traj_time)

你需要搞清的问题

  • 为什么禁止回退查找

  • 为什么不用 std::lower_bound（或只在非 realtime 路径用）

如果不这样写的工程风险

  • O(log N) 在 RT loop 中仍然不可控

  • 时间回拨 → index 来回跳 → 插值不连续

  • cache miss → 周期抖动

4️⃣ 插值前的“区间合法性确认”

要读的逻辑块

  • prev_point / next_point 的选取

  • 相邻点 time_from_start 的单调性假设

你需要搞清的问题

  • controller 对 trajectory 的 最小合法假设

  • 哪些非法轨迹不会在这里修复，而是直接失败

如果不这样写的工程风险

  • 重复时间戳 → 除零

  • 非单调时间 → 速度瞬时爆炸

  • 控制器 silently 执行错误轨迹

5️⃣ 插值公式：为什么是“线性 + 最少字段”

要读的逻辑块

  • position interpolation

  • velocity / acceleration 是否参与

  • ratio = (t - t0) / (t1 - t0)

你需要搞清的问题

  • 为什么 controller 不信任高阶插值

  • 为什么 position 是第一公民

如果不这样写的工程风险

  • 高阶插值 → 数值不稳定

  • 依赖 velocity → 上游规划误差被放大

  • 不同 joint 插值不一致 → 结构性抖动

6️⃣ sample() 的输出语义（非常容易被忽略）

要读的逻辑块

  • sample 返回的是 “期望状态快照”

  • 而不是 trajectory 的某个点引用

你需要搞清的问题

  • 为什么必须是 copy，而不是 reference

  • 为什么不能延迟计算

如果不这样写的工程风险

  • realtime buffer 被并发修改

  • 同一周期读到不一致 joint 状态

  • UB（未定义行为）直接进硬件

7️⃣ 为什么这些逻辑“只存在于 controller，而不是 MoveIt”

你需要思考的问题

  • MoveIt trajectory 为什么不能直接下发执行

  • 为什么 sampling 必须在 controller 内

如果不这样写的工程风险

  • planner time 与 controller time 不一致

  • 无法处理执行级 jitter

  • 控制器变成“被动播放器”

8️⃣ 最终工程结论（一句话）

trajectory sampling 的本质不是插值算法，而是：
在不可信时间、不可信轨迹、不稳定周期下，
给硬件一个“永远合理”的期望状态。

总结

“插值必须这样写”的工程风险一次性收束

现在我们把 interpolate / sample 这一整套写法，压缩成你要求的格式：

每一段代码回答一个问题：
“如果不这样写，会出现什么执行风险？”

① 时间 clamp（<0 / >end）

  • 不这样写：ratio 越界 → 外推 / 末端继续动

  • 执行风险：反向抽动、抖动、无法收敛、误判完成

② forward-only 区间推进

  • 不这样写：区间来回跳、边界粘住

  • 执行风险：微不连续 → 关节噪声 / 末端抖

③ <= 边界吸收

  • 不这样写：边界抖动 → ratio 在 1/0 附近来回

  • 执行风险：末端 hunting

④ 退化区间直接返回点

  • 不这样写：除零 / epsilon 放大

  • 执行风险：瞬时跳变（最危险）

⑤ joint 独立插 + 同一 ratio

  • 不这样写：关节不同步

  • 执行风险：末端轨迹变形、容差异常

⑥ position-first，vel/acc 保守

  • 不这样写：错误前馈

  • 执行风险：抖动、冲击、硬件保护

Block 6+ 的最终一句话（工程级）

trajectory sampling 的全部设计，
本质是在不可信时间、不完美轨迹和严格 realtime 约束下，
保证 controller 在每个周期只输出一个“永远可执行、不爆炸”的状态快照。
