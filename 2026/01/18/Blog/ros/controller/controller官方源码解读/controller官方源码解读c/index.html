学习大纲

0. C 段在 update() 里的位置与输入输出边界

  • C 段“接到”的东西：

    • 当前状态（joint state：pos/vel）

    • 期望状态（来自轨迹采样结果：desired pos/vel）

    • 容差配置（state/goal tolerances + time tolerances）

  • C 段“能做”的事情：

    • 判定是否越界（position/velocity）

    • 选择失败策略（abort / hold / stop 的路径分叉）

    • 产出状态与结果码（给 action/server/上游）

1. 容差参数体系：到底在比较什么

  • state tolerance（运行中容差）：轨迹执行过程中，每个周期都可能检查

  • goal tolerance（到点容差）：轨迹末端/到达目标时检查

  • goal_time_tolerance（时间容差）：到达末端允许的“超时窗口”

  • 工程含义：

    • state tolerance 保护“过程中不失控”

    • goal tolerance 保护“最终停得住且停得准”

    • time tolerance 保护“不要无限拖延导致上层误判已完成”

三层容忍度排序

  • 宽松到严格
    • state tolerance → 过程可控性
    • goal tolerance → 终点精度
    • goal_time_tolerance → 时间底线（防止无限等待）

2. position / velocity 容差判定逻辑（逐类条件分支）

2.1 position error 的计算与比较对象

  • error 形式（一般就是 abs(actual - desired)，逐 joint）

  • 分支：

    • 若 configured tolerance 缺失/为0/为NaN 的处理（常见是：缺省策略或视为不检查/严格检查，源码会有明确分支）

    • continuous joint / wrap-around（如果源码有处理，会是另一层风险控制：避免角度跳变误判）

2.2 velocity error 的计算与比较对象

  • error 形式（abs(actual_vel - desired_vel) 或 abs(actual_vel) 在末端 hold 时）

  • 关键分支：

    • 执行中比较 actual vs desired

    • 末端/保持可能比较 actual 是否“足够接近 0”（因为 hold 目标是停稳）

2.3 “越界”定义：单关节越界 vs 全部越界

  • 通常逻辑：任意 joint 越界就算 failure（工程上更保守）

  • 风险回扣：单轴失控足以造成机构碰撞/过载

3. 什么时候触发 abort / hold / stop（按决策树拆）

这一节是 C 段核心：判定结果 → 状态机转移 → 输出给上游

3.1 运行中越界（state tolerance violated）

  • 条件块：if (state_error > state_tolerance)（逐 joint）

  • 典型后果分支：

    • abort：对 action 报失败，停止继续跟踪轨迹

    • 可能伴随 hold position（进入安全保持模式，而不是继续追）

3.2 到达末端后的越界（goal tolerance violated）

  • 条件块：if (is_last_point && goal_error > goal_tolerance)

  • 后果分支：

    • 若仍在 goal_time_tolerance 内：允许继续尝试收敛（短时间“补偿到位”）

    • 若超出 goal_time_tolerance：abort（否则上游会永远等）

3.3 超时（goal_time_tolerance exceeded）

  • 条件块：if (time_since_end > goal_time_tolerance)

  • 后果分支：abort + 进入 hold/stop（看源码如何做“停止输出”或“保持输出”）

3.4 stop 的触发（外部请求 / 控制器内部自保）

  • 外部 stop：action cancel / controller switch / deactivate

  • 内部 stop：异常严重时直接停止（比 hold 更激进）

  • 风险回扣：stop 是“断输出”，hold 是“安全保持输出”（两者的风险模型不同）

4. 为什么 controller 必须“宁可中断，也不硬执行”（风险驱动解释）

把每条容差检查都回扣到“真实执行风险”：

  • 跟踪误差持续变大 → 通常不是“再追一追就好”，而是：

    • 关节饱和（速度/加速度/力矩限制触顶）

    • 摩擦/负载变化导致模型失配

    • 控制回路或 state feedback 延迟/抖动

    • 轨迹时间戳与实际时间不同步（上层时间结构被破坏）

  • 硬执行的代价：

    • 机构碰撞、夹伤、过流、减速器冲击

    • 轨迹越追越偏，最后突然“回拉”造成冲击

  • 因此 C 段策略本质是：

    • 识别“系统已不再可控/不再可信跟踪”

    • 尽早切到“可预测的安全状态”（abort + hold/stop）

5. 异常如何向上反馈到 Action / MoveIt（只讲接口语义，不讲 MoveIt 内部）

  • controller 内部把状态归因到 action 层的结果：

    • SUCCEEDED：在容差内完成

    • ABORTED：容差越界 / 超时

    • CANCELED：外部取消

  • 同时会发布/更新：

    • FollowJointTrajectory 的 result + error_code（以及可能的 error_string）

    • feedback（当前误差、当前点索引等——是否包含取决于实现）

  • MoveIt 侧你已经知道连接点：

    • MoveItPy 发目标 → action server → controller

    • C 段的判定直接决定 MoveIt 看到的“执行成功/失败”，进而影响是否 replan / 报错 / 停止任务流

6. 我们接下来怎么学（按源码逐块走）

  • 你把 C 段的源码贴出来（或至少贴 3～5 个关键 if/else 块）

  • 我们按顺序做三件事：

    1. 每个条件在保护什么风险

    2. 触发后状态机如何变化（abort/hold/stop）

    3. 对 action result 的映射是什么

controller详解

c段的使命

  • 只要发现“继续追这条轨迹，比停下来更危险”，就立刻中断
    • state tolerance 是 “执行中”就要查
    • goal tolerance 是 “终点”再查
    • goal_time_tolerance 是 “再给你一点时间，但不无限”
  • 除了加速度和速度等系统因素出现，还要防止轨迹不可信

5句话标准工程模板总结

1️⃣ controller 在每个 update 周期先基于时间采样得到 desired state。
2️⃣ C 段将当前真实 joint state 与 desired state 对比，计算 position/velocity 等误差。
3️⃣ 若在轨迹执行过程中任一 joint 超出 state tolerance，则认为轨迹不再可信，立即中断轨迹跟踪。
4️⃣ 若到达末端但未满足 goal tolerance，则在 goal_time_tolerance 窗口内允许继续收敛，超时则判失败。
5️⃣ 执行失败通过 Action 上报 aborted，同时切换到底层的 hold 状态；执行成功则上报 success 并进入 success point。
