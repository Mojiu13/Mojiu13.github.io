a段代码

A 段在 update() 中的总体定位

A 段 =「控制器每个周期的“调度与换轨逻辑”」
它解决的是：
“这一控制周期，到底该执行哪一条轨迹？用什么参数？是否切换？”
——而不是“怎么算误差”“怎么发指令”。

一、A 段在整个控制系统里的角色

站在 MoveIt → ros2_control → joint_trajectory_controller 的角度：

👉 A 段 = RT 线程里唯一允许“看新目标、换轨”的地方

二、A1：速度缩放（scaling interface）——不是轨迹逻辑，但很关键

工程视角解读

  • 这是 工业控制器必备能力

  • 典型场景：

    • 安全 PLC / teach pendant 调速

    • 急停后慢速恢复

  • 重要点：
    👉 速度缩放不改变轨迹点本身
    👉 它只作为后续采样/时间推进的“乘数”

这一步只是读取状态，不涉及 active / pending。

三、A2：动态参数更新（仍然是“调度前准备”）

关键工程事实

  • 参数更新是 RT-safe 的

  • 但：

    • 不会立刻换轨

    • 只是影响后续轨迹执行行为

👉 在控制器设计里：
“参数 ≠ 控制状态”

四、A3：active goal 的 RT 读取（极关键）

const auto active_goal = *rt_active_goal_.readFromRT();

这是你必须刻在脑子里的点

  • rt_active_goal_ 是 realtime buffer

  • Non-RT（Action 回调） 会写

  • RT（update） 只读

👉 RT 线程永远不直接操作 Action goal

这是 ROS2 控制器的铁律。

五、A4：新轨迹是否要接管？（active / pending 管理核心）

两条轨迹来源

名称 含义
current_trajectory_ 当前正在执行/准备执行的轨迹
new_trajectory_msg_ Non-RT 线程刚收到的新轨迹

核心判断条件（你一定要读懂这行）

翻译成人话：

只有在以下条件同时满足时，才允许换轨：

1️⃣ 新轨迹 ≠ 当前轨迹
2️⃣ 不存在“pending 但还没激活”的 goal

为什么要有 pending？

工程原因（非常重要）：

  • Action goal 的状态迁移是 异步的

  • Non-RT 线程里：

    • goal 已收到

    • 但 timer / accept / activate 还没走完

  • 如果此时 RT 线程抢先换轨：

👉 Action 状态机会乱套

所以：

pending = “已收到但不允许 RT 线程执行”

六、A5：真正的“轨迹接管”发生在这里

fill_partial_goal(*new_external_msg); sort_to_local_joint_order(*new_external_msg); current_trajectory_->update(*new_external_msg);

每一步在干什么（工程视角）

1️⃣ fill_partial_goal

  • MoveIt 可能只给：

    • positions

    • 不给 velocities / accelerations

  • 控制器要补齐，否则后面没法 sample

👉 这是控制器的“防御性设计”

2️⃣ sort_to_local_joint_order

  • MoveIt joint 顺序 ≠ controller 内部 joint 顺序

  • controller 永远以自身接口顺序为准

👉 这是工业级 controller 的底线原则

3️⃣ current_trajectory_->update

⚠️ 非常重要一句：

此时还没有“执行”轨迹

只是：

  • 把新轨迹

  • 转换成 controller 内部的 Trajectory 对象

  • 标记为：“尚未 sample”

七、A6：更新当前关节状态（为采样做准备）

state_current_.time_from_start = 0; read_state_from_state_interfaces(state_current_);

注意这个设计

  • time_from_start = 0

  • 说明：

    • 这是 “当前时刻的真实关节状态”

    • 不是轨迹里的 state

👉 后面如果需要 从当前状态平滑衔接新轨迹，就靠它。

八、A7：是否存在 active trajectory？

if (has_active_trajectory())

active trajectory 的含义（很精确）

不是：Action active
而是：controller 内部有一条“可执行轨迹”

九、A8：首次采样前的“轨迹对齐逻辑”（A 段的终点）

if (!current_trajectory_->is_sampled_already()) { first_sample = true; if (params_.interpolate_from_desired_state) { if (last_commanded_time_ ~ 0) last_commanded_time_ = time; current_trajectory_->set_point_before_trajectory_msg( last_commanded_time_, last_commanded_state_, joints_angle_wraparound_); } else { current_trajectory_->set_point_before_trajectory_msg( time, state_current_, joints_angle_wraparound_); } }

这是 A 段最“控制工程”的地方

你可以把它理解为：

新轨迹的“起跑姿势”怎么选？

两种模式：

模式 含义
interpolate_from_desired_state = true 从上一次命令状态平滑接
false 从当前真实状态直接接

👉 这是为了处理：

  • 控制延迟

  • 低频 / 高频 controller 混合

  • 外部干扰

十、用一句话总结 A 段（你以后写博客可以直接用）

A 段负责在 RT update 中完成“轨迹调度”：
安全地接收新轨迹、处理 active / pending、
将 Action 目标转化为 controller 内部可采样的 Trajectory，
并确定轨迹的起始对齐方式。
